1. Introduction and Problem Statement
On a modern university campus, managing facilities like dorms, lecture halls, computer labs, and office spaces is a huge task. Currently, the university relies on manual ways to report facilities issues, including phone calls, paper forms, WhatsApp messages, and walk-ins to the maintenance office.
This setup has created several major problems:
Things Get Lost:
 WhatsApp messages are easily buried, and paper slips get misplaced. Without a central database, requests frequently fall through the cracks.
Long Delays: 
It takes a long time to write a complaint from a student to the right maintenance officer. There is no clear way to see what issues are urgent.
No Accountability: 
Administrators cannot track which technician has been assigned to a job, how long they took to fix it, or whether the work was actually completed to a good standard.
Frustrated Users:
 Students and staff submit issues but have no idea if anyone is working on them. They receive no feedback unless they follow up in person.
To solve this, we developed UniMaintain, a full-stack web portal designed to make reporting and resolving campus maintenance requests simple, transparent, and organized.
2. System Objectives
The goal of this project is to replace the old manual system with an easy-to-use digital platform:
Intuitive Reporting: Give students and staff a clean interface to report faults (like plumbing, electricity, or internet issues) with details, location, and photos.
Role-Based Workflows: Ensure that students, maintenance officers, and administrators see dashboards customised to their specific roles.
Fast Task Assignment: Let administrators see new issues immediately and assign them to the right officers with a few clicks.
Full Transparency: Maintain an audit log for every request so anyone can check its history  when it was submitted, who was assigned, and when work started.
Real-Time Updates: Use WebSockets to push live status changes to connected users without requiring a page refresh.
Performance Tracking: Provide a dedicated Reports page with chart visualisations, date range filtering, and CSV export so administrators can spot recurring faults and track resolution times.
3. Requirement Analysis
The platform is designed around four distinct user roles with specific workflows:
Functional Requirements
User Authentication:
Everyone must be able to create an account and sign in securely. The system uses their login credentials to identify their role (Student, Staff, Officer, or Admin).
Student/Staff Workflow:
Submit maintenance complaints by filling out a form specifying the title, description, category (e.g., HVAC, plumbing, internet), priority level, location, and uploading photos
View a list of their own submitted requests and monitor their live status
Cancel pending requests or acknowledge resolved ones
Receive in-app notifications when tickets are updated or resolved
 Maintenance Officer Workflow:
View a dashboard of tasks assigned to them with urgency indicators
Update the status of assigned tasks (mark as &quot;In Progress&quot; when starting, &quot;Resolved&quot; when finished)
Use inline Start Work / Mark Resolved action buttons directly in the task list
Administrator Workflow:
View a dashboard showing statistics (total open tickets, issues sorted by priority and category)
Assign unassigned tickets to specific officers from the request detail panel or requests table
Manage users , invite new users, edit roles/departments, activate or deactivate accounts
View a Reports page with date-range filtering, officer performance table, and CSV export

4. Technologies Used
Frontend:
 Built using React 18 and TypeScript for a fast, component-driven user interface. Styling is written using Tailwind CSS (v4) for clean, responsive layouts. State management is handled client-side. Icons are rendered using Lucide React, charts are built with Recharts, and the date range picker uses react-day-picker.
Backend:
 Developed with Node.js and the Express framework in TypeScript. It provides a RESTful API with authenticated endpoints protected by JWT middleware. File uploads are handled via Multer. Real-time communication is implemented with Socket.io, which emits events (`request:new`, `request:updated`, `request:assigned`) whenever data changes.
// backend/src/server.ts
import express from &quot;express&quot;;
import { authRouter } from &quot;./routes/auth&quot;;
import { requestRouter } from &quot;./routes/requests&quot;;
import { authMiddleware } from &quot;./middleware/auth&quot;;
const app = express();
app.use(express.json());
app.use(&quot;/api/auth&quot;, authRouter);
app.use(&quot;/api/requests&quot;, authMiddleware, requestRouter);
API Documentation:
 Full interactive documentation is generated from JSDoc annotations using swagger-jsdoc and served via swagger-ui-express at `/api/docs`.
Database:
 Uses a PostgreSQL relational database hosted on Supabase. A relational database was chosen because the system relies on structured relationships:
Users to Requests: One-to-many relationship (a user submits many requests; an officer is assigned many requests).
Requests to Audit Logs: One-to-many relationship (each request has a detailed timeline of events).
Requests to Attachments: One-to-many relationship (each request can have multiple uploaded images).
Users to Notifications: One-to-many relationship  (each user has their own notification feed)
CREATE TABLE users (
  id SERIAL PRIMARY KEY,
  name TEXT NOT NULL,
  email TEXT UNIQUE NOT NULL,
  role TEXT NOT NULL CHECK (role IN ('student','staff','officer','admin')),
  password_hash TEXT NOT NULL,
  created_at TIMESTAMP WITH TIME ZONE DEFAULT now()
);
CREATE TABLE requests (
  id SERIAL PRIMARY KEY,
  user_id INTEGER NOT NULL REFERENCES users(id),
  category TEXT NOT NULL,
  priority TEXT NOT NULL,
  status TEXT NOT NULL,
  title TEXT NOT NULL,
  description TEXT NOT NULL,
  created_at TIMESTAMP WITH TIME ZONE DEFAULT now()
);
5. API Documentation
The backend exposes a structured JSON API to facilitate communication between the React frontend and the server. The following endpoints are protected by 
Bearer
 JWT tokens to ensure that users can only access data relevant to their role.&quot;
Full interactive documentation is available at:  
https://unimaintain-backend.onrender.com/api/docs
The backend exposes a structured JSON API. All protected endpoints require a `Bearer` JWT token in the `Authorization` header.
Example login request:
curl -X POST https://unimaintain-backend.onrender.com/api/auth/login \
  -H &quot;Content-Type: application/json&quot; \
  -d '{&quot;email&quot;:&quot;p.damilola@university.edu&quot;,&quot;password&quot;:&quot;password123&quot;}'
Example authenticated request:
curl -H &quot;Authorization: Bearer &lt;JWT_TOKEN&gt;&quot; \
  https://unimaintain-backend.onrender.com/api/requests
Authentication (`/api/auth`)
`POST /api/auth/register` — Creates a new account and returns a JWT token
`POST /api/auth/login` — Verifies credentials and returns a JWT session token
`GET /api/auth/me` — Fetches the currently authenticated user profile
Example frontend login call:
const response = await fetch(&quot;/api/auth/login&quot;, {
  method: &quot;POST&quot;,
  headers: { &quot;Content-Type&quot;: &quot;application/json&quot; },
  body: JSON.stringify({ email, password }),
});
const data = await response.json();
localStorage.setItem(&quot;unimaintain_token&quot;, data.token);
Maintenance Requests (`/api/requests`)
`GET /api/requests` — Fetches requests (Students see only theirs; Officers see assigned tasks; Admins see all). Supports `status`, `category`, `priority`, `search`, `page`, and `limit` query params
`POST /api/requests` — Submits a new ticket with optional file uploads (multipart/form-data)
`GET /api/requests/:id` — Returns full details, attachments, and audit log timeline of a ticket
`PUT /api/requests/:id/status` — Updates status (Officers: in_progress/resolved; Admins: closed/cancelled)
`PUT /api/requests/:id/assign` — (Admin only) Assigns a ticket to a maintenance officer
`GET /api/requests/stats` — (Admin only) Fetches aggregated totals for dashboard charts
Users (`/api/users`)
`GET /api/users` — (Admin only) Lists all registered users
`GET /api/users/officers` — (Admin only) Lists active officers for the assign dropdown
`PUT /api/users/:id/toggle` — (Admin only) Activates or deactivates a user account
Notifications (`/api/notifications`)
`GET /api/notifications` — Lists the current user's notifications (newest first, max 50)
`PUT /api/notifications/:id/read` — Marks a single notification as read
`PUT /api/notifications/read-all` — Marks all notifications as read

Keep-Alive Strategy
A GitHub Actions workflow (`.github/workflows/keep-alive.yml`) runs on a cron schedule every 10 minutes. It pings both the Render backend (`/api/health`) and the Supabase REST API to prevent inactivity-based shutdown on free tiers.

9. Challenges Encountered and Solutions
Large Single-File Codebase
As features were added, `App.tsx` grew to over 4,100 lines, triggering editor warnings. The solution was to extract types (`src/types/`), mock data (`src/data/`), utility functions (`src/lib/utils.ts`), and session helpers (`src/lib/session.ts`) into dedicated modules. This reduced `App.tsx` to ~3,700 lines with clear separation of concerns.
Session Persistence Across Logouts
Users were losing their changes (e.g., profile edits) when they logged out and back in. The root cause was a stale closure in `handleLogout`  ; it captured an outdated `users` state value. The fix used a `useRef` (updated synchronously after every mutation) so the logout handler always writes the latest data to storage, never a stale snapshot.
Demo Mode Data Isolation
New registrations were seeing mock data belonging to demo users. This was fixed by isolating new accounts (empty requests on login) while demo users load shared state from `localStorage`. A `DEMO_USER_IDS` whitelist separates the two paths in `handleLogin`.
Render Free Tier Inactivity
The Render backend spins down after 15 minutes of inactivity, causing a ~50-second cold start on the next request. The solution was a GitHub Actions cron job that pings the backend every 10 minutes, keeping it permanently warm.
recharts SVG Duplicate Key Warnings
Multiple charts on the same page caused React warnings about duplicate SVG keys. This was resolved by replacing all `BarChart` components in the Admin Dashboard with pure CSS/Tailwind visualisations, which eliminated the recharts internal key generation issue entirely.
Database Schema Gaps
After adding the Staff role and Cancelled status on the frontend, the backend PostgreSQL CHECK constraints rejected these values. The schema was updated to include `'staff'` in the role constraint and `'cancelled'` in the status constraint, and the `validTransitions` map in the status controller was extended accordingly.
Monorepo Testing Environment Conflicts
During development, running tests for the backend caused Vitest to recursively check parent directories and attempt to load the root 
vite.config.ts
, which required browser-specific 
jsdom
 settings. This resulted in Node.js test failures. The solution was to create a dedicated 
backend/vite.config.ts
 configuration and explicitly define 
include
 paths to isolate the frontend browser tests from the backend API tests.

1. Authentication Middleware (Backend)
This middleware intercepts incoming requests, verifies the JSON Web Token, and attaches the authenticated user to the request object for role-based access control.
import { Request, Response, NextFunction } from &quot;express&quot;;
import jwt from &quot;jsonwebtoken&quot;;
export interface JwtPayload {
  id: number;
  email: string;
  role: &quot;student&quot; | &quot;officer&quot; | &quot;admin&quot;;
  name: string;
}
// Extend Express Request to carry the decoded user
declare global {
  namespace Express {
    interface Request {
      user?: JwtPayload;
    }
  }
}
export function authenticate(req: Request, res: Response, next: NextFunction): void {
  const authHeader = req.headers.authorization;
  if (!authHeader || !authHeader.startsWith(&quot;Bearer &quot;)) {
    res.status(401).json({ error: &quot;No token provided&quot; });
    return;
  }
  const token = authHeader.split(&quot; &quot;)[1];
  try {
    const decoded = jwt.verify(token, process.env.JWT_SECRET || &quot;secret&quot;) as JwtPayload;
    req.user = decoded;
    next();
  } catch {
    res.status(401).json({ error: &quot;Invalid or expired token&quot; });
  }
}
2. Time-Based Greeting Logic (Frontend)
A utility function used across all role dashboards to calculate a personalized greeting based on the local system time.
export function getGreeting(): string {
  const hour = new Date().getHours();
  if (hour &lt; 12) return &quot;Good morning&quot;;
  if (hour &lt; 17) return &quot;Good afternoon&quot;;
  return &quot;Good evening&quot;;
}
